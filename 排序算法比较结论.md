# 排序算法性能比较（数据结构综合实践作业）

## 项目简介
本项目使用 Java 实现了 **8 种经典排序算法**，并对比它们在 **升序、降序、随机** 三种不同输入分布下的性能表现。 
数组长度设计为 **100 万**，同时为了演示效果，对时间复杂度为 O(n²) 的慢速算法在逆序和随机情况下自动降低数据量，避免长时间等待。  
所有核心数据结构与排序逻辑均自行实现，未调用 Java 标准库中的排序或集合工具，符合老师要求。

## 成员分工
- **骆帅**  
  - 数组生成逻辑（升序、降序、随机数组）  
  - 数组文件读写工具类（`ArrayFileUtil`）  
  - 主类 `Main.java`（测试框架、数据加载、性能计时、结果输出）  

- **梁可**  
  - 八种排序算法的实现与优化：  
    `SelectionSort`、`Insertion`、`BubbleSort`、`ShellSort`、  
    `MergeSort`、`QuickSort`、`HeapSort`、`RadixSort`

## 1. 数组生成与存储

### 1.1 生成三种数组
- **升序数组**：1, 2, 3, …, 1,000,000  
- **降序数组**：1,000,000, 999,999, …, 1  
- **随机数组**：每个元素为 `[1, 1,000,000]` 范围内的随机整数

生成代码位于 `Main.java` 中的 `generateAscending`、`generateDescending`、`generateRandom` 方法，使用普通的循环与 `java.util.Random` 实现。

### 1.2 文件持久化
为了节省每次运行时的生成时间，程序引入 `ArrayFileUtil` 工具类：
- 首次运行时，将三个数组以二进制格式写入文件（`ascending.dat`、`descending.dat`、`random.dat`）。
- 文件格式：先写入数组长度（`int`），再依次写入所有元素。
- 再次运行时，若检测到这三个文件已存在，则直接加载，跳过生成过程。

这一设计特别适用于答辩演示场景，能够快速启动测试，无需每次等待 100 万数据生成。

## 2. 实现的 8 种排序算法

每种算法封装为一个独立的类，提供统一的静态方法 `public static void sort(int[] a)`。

### 2.1 简单选择排序 – `SelectionSort`
- **原理**：每一轮从未排序区间选择最大元素，放到已排序区间的末尾。
- **时间复杂度**：任何情况下均为 O(n²)。
- **实现特点**：交换次数少，但比较次数固定，性能最差的一类。

### 2.2 直接插入排序 – `Insertion`
- **原理**：将数组分为已排序和未排序两部分，依次将未排序元素插入到已排序部分的合适位置。
- **时间复杂度**：最好 O(n)（已升序），最坏/平均 O(n²)。
- **实现特点**：采用**迭代**方式而非递归，避免百万数据量导致栈溢出。

### 2.3 冒泡排序 – `BubbleSort`
- **原理**：重复遍历数组，比较相邻元素并交换，使较大元素逐步“浮”到末尾。
- **时间复杂度**：最好 O(n)（已升序），最坏/平均 O(n²)。
- **实现特点**：增加 `swapped` 标志，若某一轮无交换则提前终止，提升有序情况下的速度。

### 2.4 希尔排序 – `ShellSort`
- **原理**：使用递减的增量（gap）对数组进行分组插入排序，最后 gap=1 时即为标准插入排序，但由于前期的大步长使元素基本有序，最终效率很高。
- **时间复杂度**：与增量序列有关，此处采用 `gap = n/2, n/4, ..., 1`，时间复杂度约为 O(n log² n) 或 O(n^1.3) 左右。
- **实现特点**：逻辑紧凑，利用插入排序对部分有序数据高效的特点。

### 2.5 归并排序 – `MergeSort`
- **原理**：分治策略，将数组递归分成两半，分别排序后合并两个有序序列。
- **时间复杂度**：稳定 O(n log n)，需要 O(n) 额外空间。
- **实现特点**：自顶向下递归实现，合并时使用临时数组暂存结果。

### 2.6 快速排序 – `QuickSort`
- **原理**：每次选择一个基准元素，将数组分为小于基准和大于基准的两部分，然后递归排序。
- **时间复杂度**：平均 O(n log n)，最坏 O(n²)。
- **实现特点**：使用**三数取中法**选择基准，将 `low`、`mid`、`high` 三个位置的中值作为基准，有效避免在有序数组上退化为最坏情况。

### 2.7 堆排序 – `HeapSort`
- **原理**：先将数组构建成最大堆，然后反复将堆顶（最大值）与末尾元素交换，并缩小堆范围，对新堆顶下滤，最终得到有序序列。
- **时间复杂度**：稳定 O(n log n)，原地排序。
- **实现特点**：下滤（`down`）操作采用递归，建堆从最后一个非叶子节点开始。

### 2.8 基数排序 – `RadixSort`
- **原理**：LSD（最低位优先）基数排序，从个位开始，对每一位进行稳定的计数排序，最终得到整体有序。
- **时间复杂度**：O(d·(n + k))，其中 d 为位数（此处最多 7 位），k 为基数（十进制为 10），近似线性时间。
- **实现特点**：仅适用于非负整数，正好匹配本项目的数据范围（1 ~ 1,000,000）。

## 3. 测试框架 – Main 类

`Main` 类是程序的入口，负责：

1. **加载或生成数据**：通过静态块在程序启动时检查数据文件是否存在，存在则加载，否则生成并保存。
2. **性能测试**：
   - 提供 `testSort` 方法，接收算法名称、原始数组和数组类型标签。
   - **答辩友好设计**：对于 O(n²) 慢速算法（选择排序、直接插入排序、冒泡排序），在**降序**和**随机**数据上自动使用 **1 万** 数据量进行测试，避免演示时长时间等待（原本百万数据可能需要数十分钟）。
   - 高效算法（希尔、归并、快速、堆、基数）始终使用 **100 万** 数据量测试。
   - 每次排序前均复制原始数组，保证不同算法之间互不影响。
   - 使用 `System.currentTimeMillis()` 计时，输出格式化的表格。
3. **结果输出**：控制台打印包含算法名、数组类型、数据量、耗时的清晰报表，也可通过重定向保存为文本文件。

### 分析结论
1. **O(n²) 算法的严重缺陷**  
   - 即使数据量仅有 **1 万**，选择、插入、冒泡排序在逆序/随机情况下也需要几十毫秒到上百毫秒。  
   - 当数据量增加到 **100 万** 时，它们的耗时会达到数百甚至上千秒，完全不可接受。  
   - 插入排序和冒泡排序在升序情况下得益于提前终止机制，可达到 O(n) 近乎线性，但对数据分布极其敏感。

2. **O(n log n) 算法的统治地位**  
   - 快速排序、归并排序、堆排序、希尔排序、基数排序在 **100 万** 数据量下均能在 **0.1 秒左右甚至更短** 内完成。  
   - 基数排序因其近似线性的特性表现最优，但适用范围受限于整数数据。  
   - 快速排序通过三数取中优化，成功规避了有序数据上的最坏情况，性能稳定高效。

3. **工程实践启示**  
   - 数据量越大，算法选择越关键。O(n²) 算法仅适合极小规模或基本有序的数据。  
   - 实际应用中应优先选择 O(n log n) 的通用排序算法，并根据数据特性考虑基数排序等线性方法。

## 5. 白盒测试与正确性验证

为确保每个排序算法实现完全正确，项目汲取了软件工程中的知识，进行了系统化的白盒测试，从代码逻辑层面覆盖各种输入情况（本次测试由DeepSeek进行完成）。

### 5.1 测试策略
- **基本功能测试**：验证排序能否将输入数组正确排为升序。
- **边界测试**：空数组、单元素数组、两个元素数组。
- **典型数据测试**：已排序、逆序、随机、含重复元素、全部相同元素。
- **规模压力测试**：10,000 到 100,000 随机数据，验证大数据量下稳定性。
- **语句与分支覆盖**：设计测试用例触发每个条件分支（如 `if`、`while` 循环结束条件、提前退出标志等）。

### 5.2 测试用例设计
对每个排序算法，至少包含以下用例（以 `SelectionSort` 为例）：
- 空数组 → `[]` 应保持为空。
- 单元素 → `[5]` 不变。
- 两个元素升序 → `[1,2]` 保持 `[1,2]`。
- 两个元素降序 → `[2,1]` 变为 `[1,2]`。
- 已升序数组 → `[1,2,3,4,5]` 结果不变。
- 已降序数组 → `[5,4,3,2,1]` 变为升序。
- 含重复元素 → `[3,1,2,3,1]` 变为 `[1,1,2,3,3]`。
- 全部相同 → `[7,7,7]` 保持 `[7,7,7]`。
- 随机数组（长度 100）→ 排序后与标准 `Arrays.sort()` 结果一致（仅在测试时临时用于参考比对，最终代码本身不使用库排序）。

### 5.3 测试执行
采用手工编写测试代码的方式进行，未引入 JUnit 等外部框架，完全自行实现断言逻辑。测试代码片段示例：

## 6.1 结论

1. **时间复杂度是决定排序算法效率的核心因素**
   - O(n²) 的算法（选择排序、直接插入排序、冒泡排序）在处理大规模数据时性能极差，即使仅 1 万条数据也需要数十毫秒，在 100 万数据量下甚至需要数百秒，完全无法满足实际应用需求。
   - O(n log n) 的算法（希尔排序、归并排序、快速排序、堆排序）在百万数据量下均能在 0.1 秒左右完成，表现出稳定的高效性。基数排序凭借近似 O(n) 的线性复杂度，速度更优，但适用范围受数据特征限制。
2. **数据分布对算法性能影响显著，尤其对 O(n²) 算法**
   - 直接插入排序和冒泡排序在升序数据下可接近 O(n)，但面对降序或随机数据时退化为严格的 O(n²)。
   - 简单选择排序无论数据如何分布均为 O(n²)，效率始终低下。
   - 而 O(n log n) 算法（如归并排序、堆排序）对数据分布不敏感，性能稳定；快速排序通过三数取中优化后，也能消除有序数据带来的最坏情况。
3. **算法优化具有关键实用价值**
   - 三数取中法有效避免了快速排序在有序或逆序数据上的退化，使原本可能栈溢出或极慢的情况变为平均性能。
   - 冒泡排序的提前终止标志（swapped）在数据基本有序时将比较次数降到线性级。
   - 迭代实现直接插入排序避免了递归带来的栈溢出风险，保证了百万数据的顺利处理。
4. **工程实现中的细节不可忽视**
   - 数据文件缓存机制显著缩短了重复实验的准备时间，体现了实用程序设计的必要性。
   - 严谨的白盒测试覆盖边界条件（空数组、单元素、重复值等）确保了算法的正确性，是软件质量的重要保障。
   - 通过统一接口和模块化设计，不同算法可无缝接入测试框架，展现了良好代码结构带来的可维护性。
5. **算法选择必须结合实际场景**
   - 对于极小规模或基本有序的数据，简单算法仍有用武之地；但在大数据时代，O(n log n) 甚至线性排序算法是首选。
   - 若数据特征满足基数排序的假设（非负整数、范围可控），应优先考虑，可获得近似线性的极致效率。
